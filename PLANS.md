# SH1107 Rendering Plan

Updated: 2026-04-09
Scope: Fix incorrect text rendering on the Waveshare Pico OLED 1.3 (SH1107) with minimal driver changes.

## Current Diagnosis

- The firmware is correctly configured for the SH1107 backend, not SSD1306.
- The logical framebuffer generated by ViviPi is correct at 128x64; the captured boot logo buffer shows the full `ViviPi` string without clipping before transport.
- The real hardware photo disproved a direct 128x64 page write. The panel is consuming a portrait-native 64x128 transport layout.
- The remaining issue is in the native SH1107 transport mapping, not in the logical framebuffer generation.
- The SH1107 backend must rotate the logical 128x64 buffer into a 64x128 native page stream; direct writes scramble the image on-device.
- Hardware calibration established the correct native SH1107 column origin for `waveshare-pico-oled-1.3` as `32`.

## Hypotheses Ranked by Likelihood

1. Native 64x128 transport is required, but the remaining page origin or scan orientation is still slightly wrong on the real panel.
2. A non-zero SH1107 column offset may be required after the restored native transport.
3. Segment remap or COM scan direction may still need inversion.
4. The application framebuffer dimensions are wrong.
5. Font size overflow is causing the clipping.

## Experiments to Validate Each Hypothesis

- Capture the pre-transport boot-logo framebuffer and confirm whether `ViviPi` is fully rendered in the logical 128x64 buffer.
- Decode the current rotated transport buffer and compare it with the logical framebuffer to see whether the driver is introducing the mismatch.
- Validate the restored rotated 64x128 transport against the real panel photo.
- Keep column origin explicit and sweep `0`, `2`, and `4` if the next hardware photo still shows a shift.
- Render a border and marker pattern in tests to confirm predictable top-left origin and no out-of-bounds clipping.

## Fix Strategy

- Keep the ViviPi logical display geometry at 128x64 to preserve the spec-defined 16x8 grid.
- Use a rotated 64x128 SH1107 transport path because the panel rejects direct 128x64 page writes.
- Keep the native column origin explicit and tune it only if the next physical photo proves a shift.
- Add regression tests for initialization geometry, page write shape, and border-safe rendering.
- Update traceability only if the new tests materially change requirement coverage.

## Verification Criteria

- `ViviPi` remains fully visible in the logical framebuffer and displays correctly on the real panel through the rotated native transport.
- SH1107 `_show()` writes the native portrait page stream expected by the panel.
- The top-left origin of the visible area is confirmed by physical photo, including any required native column offset.
- Border and marker tests plus hardware photos prove no clipping at the visible edges.
- Focused firmware display tests pass after the change.
